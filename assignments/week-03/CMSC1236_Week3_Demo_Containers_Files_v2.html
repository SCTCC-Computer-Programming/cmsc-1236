Learning Python 6e (Ch. 8–9): Collections (Sequences, Mappings, Sets) + File Basics

This demo is a structured walkthrough of key content from Learning Python (6e), Chapters 8–9. It focuses on Python’s built-in collection objects—especially lists and dictionaries—and on foundational file usage patterns. Each example is intended to be read and executed to make the behavior of these types explicit.

How this extends prior introductory coverage

Introductory coverage of lists, dictionaries, and sets typically emphasizes correct usage: create a collection, loop over it, and produce correct output. Chapters 8–9 extend that foundation by emphasizing the runtime model needed to use collection objects reliably in larger programs:

  • Collection objects store references to other objects.
  • Mutation (in-place change) and rebinding (name reassignment) have different effects.
  • Nested structures require careful thinking about copies (shallow vs deep).
  • Dictionaries behave as mappings with key-based access and controlled update patterns.
  • File I/O should follow predictable patterns to avoid failures (open/operate/close via with and basic error handling).

Outcomes

After completing this demo, you should be able to:

  • Distinguish identity from equality (is vs ==).
  • Distinguish rebinding from in-place mutation and predict the consequences of aliasing.
  • Apply core behaviors of sequence types (lists/tuples), the mapping type (dict), and set types (set/frozenset).
  • Copy nested collection objects correctly (shallow vs deep).
  • Read and write a small text file using with.
  • Apply a basic load-or-default approach for file-based data.

1) Names, objects, and references

In Python, a variable is a name that refers to an object. An object is a value stored in memory (for example, a list, a dictionary, an integer, or a string). When a name refers to an object, the name is said to be bound to that object.

This model is a basic part of programming in Python. It explains how changes to lists and dictionaries propagate through a program, how objects behave when passed into functions, and why copying and nesting can produce unexpected results if references are not understood.

Python’s built-in collection objects (lists, dictionaries, sets, and tuples) store references (links) to objects rather than copying them. As a result, the same object can be reached through more than one name.

Two operations are used throughout Python programs, and they are easy to confuse because both can occur after an assignment statement. The difference is what changes:

  • rebinding changes which object a name refers to; the original object is unchanged
  • mutation changes the contents of an existing object; any name that refers to that object will observe the change

Because Python programs frequently reuse objects through multiple names (intentionally or unintentionally), it is important to distinguish “same object” from “same value.” Two comparison operators are used for that purpose:

  • is tests identity: whether two names refer to the same object in memory
  • == tests equality: whether two objects have the same value

The following two examples establish the difference between identity and equality for collection objects.

This first example shows what it means for two names to refer to the same object. The output will confirm that a is b is true (same object) and a == b is also true (same contents).

a = [1, 2, 3]
b = a
t
print("a =", a)
print("b =", b)
print("a is b =", a is b)
print("a == b =", a == b)

Explanation:

  1. a = [1, 2, 3] creates a list object containing the integers 1, 2, and 3, and assigns the name a to refer to that list.
  2. b = a assigns the name b to refer to the same list object as a. No new list is created.
  3. print("a =", a) prints the label a = followed by the current value of a (the list object that a refers to).
  4. print("b =", b) prints the label b = followed by the current value of b (the same list object, since b refers to the same object as a).
  5. print("a is b =", a is b) prints the label a is b = followed by the result of the identity test a is b. This is True here because a and b refer to the same object in memory.
  6. print("a == b =", a == b) prints the label a == b = followed by the result of the equality test a == b. This is True here because the two values have the same contents (and in this case they are the same list object).

In the textbook and in interactive Python sessions (the shell or a notebook), it is common to write a single line that groups several values together so they are displayed at once (like is shown below). This does not change what Python computes; it only changes how the results are shown. Compared to labeled print statements, this style is less explicit, but it is more concise for quick demonstrations.

a = [1, 2, 3]
b = a

(a, b, a is b, a == b)

In this version, the expression (a, b, a is b, a == b) groups four results into a single tuple so they can be displayed together in one output. A tuple is a built-in Python sequence type that groups multiple values into a single object. It is created by writing values separated by commas; parentheses are often used for readability, but the commas are what matter.

Explanation:

  1. a = [1, 2, 3] creates a list object containing the integers 1, 2, and 3, and assigns the name a to refer to that list.

  2. b = a assigns the name b to refer to the same list object as a. No new list is created.

  3. (a, b, a is b, a == b) creates a tuple containing four values and displays it:

    1. a is the list object that a refers to.
    2. b is the list object that b refers to (the same list as a).
    3. a is b is a Boolean value that reports whether a and b refer to the same object in memory (identity). In this case it is True.
    4. a == b is a Boolean value that reports whether the two values have the same contents (equality). In this case it is True because both names refer to the same list with the same contents.

This second example shows that two different objects can still be equal in value. The output will confirm that x is y is false (different objects) while x == y is true (same contents).

x = [1, 2]
y = [1, 2]
x, y, (x is y), (x == y)

Explanation:

  1. x = [1, 2] creates a list object containing the integers 1 and 2, and assigns the name x to refer to that list.

  2. y = [1, 2] creates a second list object containing the integers 1 and 2, and assigns the name y to refer to that second list. Even though the contents match, this is a different object in memory from the one referenced by x.

  3. x, y, (x is y), (x == y) groups four values into a single tuple so they can be displayed together in one result in an interactive environment:

    1. x is the first list object (the one referenced by x).
    2. y is the second list object (the one referenced by y).
    3. x is y is a Boolean value that reports whether x and y refer to the same object in memory (identity). This is False because two separate list objects were created.
    4. x == y is a Boolean value that reports whether the two lists have the same contents (equality). This is True because both lists contain the values 1 and 2 in the same order.

2) Rebinding vs in-place mutation (lists)

Two operations can look similar but behave differently for lists:

  • nums = nums + [...] creates a new list (rebinding)
  • nums += [...] changes the existing list in place (mutation)

This matters whenever more than one name refers to the same list object.

nums = [10, 20, 30]
alias = nums

nums = nums + [40]
nums, alias, (nums is alias)
nums = [10, 20, 30]
alias = nums

nums += [40]
nums, alias, (nums is alias)

3) Lists (sequence type): slice assignment and nested list construction

3a) Slice assignment (in-place change)

Slice assignment replaces a region of a list in place.

data = [0, 1, 2, 3, 4, 5]
data[2:5] = ["a", "b"]
data

3b) Nested lists and repeated references

Using [inner] * n repeats references to the same inner object. If the inner object is mutable (such as a list), mutating one “row” mutates them all.

row = [0, 0]
grid_bad = [row] * 3
grid_bad[0][0] = 99
grid_bad

A correct construction creates independent inner lists:

grid_ok = [[0, 0] for _ in range(3)]
grid_ok[0][0] = 99
grid_ok

4) Dictionaries (mapping type): insertion order, safe access, and controlled updates

4a) Insertion order

Dictionaries preserve insertion order. Iteration follows the order keys were added; this does not imply keys are automatically sorted.

d = {"b": 2, "a": 1, "c": 3}
list(d.keys())

4b) Safe access and membership

Use membership tests and .get() when a key may be absent.

config = {"mode": "test"}
("mode" in config), ("timeout" in config)
config.get("mode"), config.get("timeout", 30)

4c) Controlled updates (copy + update)

A common pattern is to create a copy and then apply updates.

base = {"host": "localhost", "port": 8000}
override = {"port": 9000}

merged = base.copy()
merged.update(override)

base, override, merged

4d) Dictionary key views for comparisons

Dictionary keys() views support set operations such as intersection and difference.

a = {"x": 1, "y": 2, "z": 3}
b = {"y": 20, "z": 30, "w": 40}

a.keys() & b.keys(), a.keys() - b.keys()

5) Set types: uniqueness and hashability

A set is an unordered collection of unique elements. Elements must be hashable (typically immutable).

Sets are used for:

  • membership tests (in),
  • de-duplication,
  • comparisons (union/intersection/difference).
items = ["apple", "apple", "pear", "banana", "banana"]
set(items)
try:
    bad = {[1, 2], [3, 4]}   # lists are not hashable
except TypeError as e:
    e

frozenset is an immutable set type.

fs = frozenset({1, 2, 3})
fs, (2 in fs)

6) Tuples (sequence type, immutable): record-style grouping

Tuples are immutable sequences. They are appropriate when the “shape” of the data is fixed and the values should not be mutated after creation.

record = ("Ada", "Lovelace", 1815)
record[0], record[-1]
try:
    record[0] = "ADA"
except TypeError as e:
    e

7) Copying nested collection objects (shallow vs deep)

When data is nested, copying requires precision:

  • Shallow copy: copies the outer collection; inner objects remain shared
  • Deep copy: recursively copies nested objects

A shallow copy of a nested list shares inner lists:

nested = [[1, 2], [3, 4]]
shallow = nested[:]          # shallow copy

shallow[0].append(999)
nested, shallow

A deep copy creates independent nested objects:

import copy

nested = [[1, 2], [3, 4]]
deep = copy.deepcopy(nested)

deep[0].append(999)
nested, deep

8) File objects: predictable read/write patterns

8a) Writing and reading a text file

Use with to ensure the file is closed automatically.

from pathlib import Path

demo_path = Path("demo_text.txt")
demo_path
lines = ["first line\n", "second line\n", "third line\n"]

with demo_path.open("w", encoding="utf-8") as f:
    f.writelines(lines)

demo_path.read_text(encoding="utf-8")
with demo_path.open("r", encoding="utf-8") as f:
    data = f.read()

data

8b) Load-or-default (basic reliability pattern)

A baseline pattern for file-based data is:

  1. attempt to load,
  2. if the file is missing or invalid, use a default value,
  3. save later when appropriate.
import json
import datetime
from pathlib import Path

state_path = Path("demo_state.json")
default_state = {"version": 1, "notes": []}

def load_or_default(path: Path, default: dict) -> dict:
    try:
        with path.open("r", encoding="utf-8") as f:
            return json.load(f)
    except FileNotFoundError:
        return default
    except json.JSONDecodeError:
        return default

def save_state(path: Path, state: dict) -> None:
    with path.open("w", encoding="utf-8") as f:
        json.dump(state, f, indent=2)

state = load_or_default(state_path, default_state)
state["notes"].append("demo note at " + datetime.datetime.utcnow().isoformat() + "Z")
save_state(state_path, state)

state_path.read_text(encoding="utf-8")

Summary of key points

  • Names refer to objects; collection objects store references to other objects.
  • is tests identity; == tests value equality.
  • Rebinding and mutation produce different effects.
  • Lists and dictionaries are mutable; tuples are immutable.
  • Set types enforce uniqueness and require hashable elements.
  • Shallow copies share nested objects; deep copies replicate nested objects.
  • File work should follow predictable patterns using with, with basic handling for missing or invalid files.
